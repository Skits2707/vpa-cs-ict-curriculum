1. What is the IA?

The Internal Assessment is a software development project that solves a real problem for a real client. It is not a hypothetical exercise — the client must exist, the problem must be genuine, and the solution must be implemented and evaluated.

3. Criterion A: Planning (6 marks)

A.1 — The Scenario (Context)

Describe the client's organisation and their role. Explain why the client needs a software solution.

Good example:
"Ms. Zhang is the librarian at Victoria Park Academy Shenzhen. She currently manages book loans using a paper logbook, which makes it difficult to track overdue books, identify popular titles, or generate reports for the administration. She has agreed to be my client for this project."

A.2 — The Problem

State the problem clearly, in the client's own words where possible. The problem must be specific and solvable with software.

A.3 — Rationale

Explain why a computational solution is appropriate. Consider alternatives and justify your choice.

A.4 — Success Criteria

3–5 criteria that are specific, measurable, and testable.

Good example:
1. The system shall allow a book to be checked out in fewer than 30 seconds.
2. The system shall generate an overdue report automatically every Monday morning.
3. The system shall store at least 5,000 book records.
4. The system shall be usable by the librarian without technical training (intuitive interface).

A.5 — Record of Tasks

A dated log of all activities undertaken. Must show evidence of consultation with the client.

4. Criterion B: Design (6 marks)

B.1 — Record of Tasks (continued)

The record of tasks spans Criteria A and B. Show design-phase activities.

B.2 — System Overview

A diagram showing the overall structure of your solution. Include:

  • Main modules/components
  • Data storage
  • User interface overview
  • Data flow between components

B.3 — Algorithms

Design your key algorithms in pseudocode before coding. Show:

  • Input and output
  • Processing steps
  • Loops and conditions
  • Complex techniques (e.g., file handling, searching, sorting)

5. Criterion C: Development (12 marks)

C.1 — Complex Techniques

To achieve high marks, you must demonstrate at least two complex techniques. Examples include:

  • File handling (reading/writing structured data)
  • Searching or sorting algorithms
  • Use of collections/arrays of records
  • Object-oriented programming (classes, inheritance)
  • Database connectivity (SQL)
  • API integration
  • Recursive algorithms
  • Multi-threading/concurrency

C.2 — Code with Commentary

Include key sections of code with inline commentary explaining:

  • What the code does
  • Why you chose this approach
  • How it relates to your design

C.3 — Video Evidence

A 5–7 minute video demonstrating your solution. The video must:

  • Show the program running
  • Demonstrate each success criterion being met
  • Include narration explaining what is happening
  • Be clear and well-paced

6. Criterion D: Evaluation (4 marks)

D.1 — Testing Against Success Criteria

Show evidence that each success criterion was tested. Use screenshots, test data, and results.

D.2 — Client Feedback

Include direct feedback from your client. This can be:

  • Written evaluation
  • Email response
  • Meeting notes
  • Audio/video recording

D.3 — Future Improvements

Identify at least two realistic improvements that could be made with more time/resources.

8. IA Exemplar Outline (Grade 7)

Client: PE Department Head

Problem: Managing sports equipment loans, tracking condition, generating maintenance alerts

Solution: Equipment inventory system with barcode scanning, loan tracking, automated maintenance scheduling

Complex Techniques:

  • File I/O with CSV parsing (structured data handling)
  • Binary search for equipment lookup (efficient searching)
  • Object-oriented design (Equipment, Loan, User classes)
