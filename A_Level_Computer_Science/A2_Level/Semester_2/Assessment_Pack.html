2. Mock Exam 2 — Paper 1 (Theory)

Specification

  • Duration: 1 hour 45 minutes
  • Marks: 75
  • Format: Structured questions, short answers, extended responses
  • Content: AS + A2 theory topics (Data Representation, Hardware, Software, Communication, Security, AI, Ethics, Boolean Logic, Advanced Theory)
  • Conditions: Silent room, no notes, non-programmable calculator, black pen

Mark Scheme Structure

Question TypeMarksMarking Approach
1-mark recall1Correct answer only. No partial credit.
2–3 mark describe2–3One mark per valid point. Must be in context.
3–4 mark explain3–4One mark per reason + one mark for development/elaboration.
4–6 mark compare4–6Two marks per valid comparison point (similarity + difference). Structure rewarded.
6–8 mark analyse6–8Breakdown of components (2–3 marks), relationships (2–3 marks), conclusion (1–2 marks).
8–12 mark evaluate8–12Introduction (1), points with evidence (4–6), counter-argument (2–3), conclusion (1–2). Balanced argument essential.

Sample Question (Evaluate — 10 marks)

Question: Evaluate the use of biometric authentication compared to password authentication for securing a banking application.

Mark Scheme:

  • Introduction (1): Define biometric authentication and password authentication. State the context (banking security).
  • Point 1 + Evidence (2): Biometrics are more secure because they are unique to the individual and cannot be easily shared or stolen. Example: fingerprint or iris scan.
  • Point 2 + Evidence (2): Passwords are cheaper and easier to implement. No special hardware required. Users are familiar with them.
  • Point 3 + Evidence (2): Biometrics have privacy concerns. If biometric data is compromised, it cannot be changed (unlike a password). Storage of biometric data raises GDPR issues.
  • Counter-argument (2): Biometrics can be spoofed (e.g., fake fingerprints). Multi-factor authentication (password + biometric) offers the best balance.
  • Conclusion (1): For high-security banking, biometrics offer superior security but should be combined with passwords for defence in depth.

3. Mock Exam 3 — Paper 2 (Programming)

Specification

  • Duration: 2 hours
  • Marks: 75
  • Format: Structured questions requiring pseudocode, Python code, algorithm design, trace tables, and explanations
  • Content: Programming constructs, data types, subroutines, file handling, arrays, OOP, algorithms, computational thinking
  • Conditions: Silent room, no notes, non-programmable calculator, black pen, pencil for diagrams

Mark Scheme Structure

Question TypeMarksMarking Approach
Trace table3–5One mark per correct row/column. Must show all variable changes.
Pseudocode / Code4–12One mark per correct construct. Efficiency and structure rewarded. Syntax errors penalised only if logic is unclear.
Algorithm design6–10Decomposition (2), correct logic (3–4), appropriate data structures (2), efficiency (1).
Explanation of code2–4One mark per valid point. Must reference specific lines or variables.
Comparison of algorithms4–6Time complexity (2), space complexity (2), suitability for context (2).

Sample Question (Algorithm Design — 10 marks)

Question: A program stores student records in a text file. Each record contains: StudentID (string), Name (string), Score (integer). Write a program that reads the file, stores the data in an appropriate data structure, and allows the user to search for a student by ID using binary search. Output the student's name and score if found, or a suitable message if not found.

Mark Scheme:

  • Data structure (2): Array of records (or list of dictionaries) declared correctly. Fields match requirements.
  • File handling (2): File opened correctly, read line by line, parsed into fields, closed properly.
  • Binary search algorithm (4): Data sorted before search (1), lower/upper bounds initialised (1), loop with midpoint calculation (1), correct comparison and bounds adjustment (1).
  • Output (1): Correct output for found and not-found cases.
  • Structure and efficiency (1): Modular design, meaningful variable names, comments.

The Practical Programming Project is assessed across six sections, totalling 75 marks.

Detailed Criteria

SectionCriteria for Full Marks
AnalysisClear, well-researched problem statement. Comprehensive requirements (functional and non-functional). Appropriate, measurable success criteria. Evidence of stakeholder consultation.
DesignLogical decomposition into modules. Appropriate algorithms with clear pseudocode/flowcharts. Justified data structures. Class diagram (if OOP). Data dictionary.
DevelopmentFully working solution. Efficient, well-structured code. Appropriate use of programming constructs. Evidence of iterative development (Git history). Good code organisation and comments.
TestingComprehensive test plan. Normal, boundary, and erroneous data. Evidence (screenshots). All tests linked to requirements. Failed tests identified, fixed, and re-tested.
DocumentationClear technical guide (system design, data dictionary, annotated code). Usable user guide (installation, usage, screenshots, troubleshooting). Professional presentation.
EvaluationCritical review against all success criteria. Honest identification of strengths and limitations. Stakeholder feedback gathered and responded to. Maintenance plan with examples. Future enhancements suggested.

10. Post-Mock Analysis Protocol

After every mock exam, follow this protocol to maximise learning:

  1. Step 1 — Self-Assessment (Day 1): Students mark their own paper using the mark scheme. Complete a self-reflection sheet.
  2. Step 2 — Teacher Marking (Day 1–2): Teacher marks all papers using the official mark scheme. Records marks per question in the tracking spreadsheet.
  3. Step 3 — Class Analysis (Day 3): Teacher presents class-level data: average per question, common errors, high-scoring approaches.
  4. Step 4 — Individual Feedback (Day 3–4): Each student receives a personalised feedback sheet with marks, comments, and a target action plan.
  5. Step 5 — Targeted Intervention (Day 4–5): Students attend clinics, complete practice packs, or receive peer mentoring based on their gaps.
  6. Step 6 — Retest (Following Week): Students complete a short retest on their weakest area to demonstrate improvement.
